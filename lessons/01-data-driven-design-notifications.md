# Lesson 1: Data-Driven Design Through Notifications

## Goal

This lesson teaches system design from the data outward. Instead of starting with boxes and arrows, we start with product behavior, access patterns, correctness requirements, and data movement.

The system we will design is a notification platform because it touches many of the course themes:

- relational modeling
- indexes
- partitioning and sharding
- streams
- idempotency
- optimistic and pessimistic locking
- caches and derived read models
- eventual consistency
- cloud deployment tradeoffs

## Scenario

Design a notification system for a mobile app.

Requirements:

- users receive notifications
- users can mark notifications as read
- the app shows the 50 most recent unread notifications
- some users have millions of notifications
- mobile badge count should be fast
- notifications may be generated by many services
- duplicate notifications should be avoided where possible

## Part 1: Start with product questions

Before choosing Postgres, Redis, Kafka, DynamoDB, Cassandra, or anything else, answer these questions.

1. Are notifications private per-user objects, or can one notification object be shared by many users?
2. Can one event generate notifications for many users?
3. Are notifications immutable after creation?
4. Can users delete or archive notifications?
5. Does read/unread state belong on the notification row, or should it live separately?
6. How fresh does the mobile badge count need to be?
7. What duplicate behavior is acceptable?
8. How long are notifications retained?
9. Are notifications generated synchronously in the user request path, or asynchronously from events?
10. Are there users or events that can create extreme fanout?

## Part 3: First relational model

Start with a simple Postgres-centered design.

```sql
CREATE TABLE notifications (
  id BIGSERIAL PRIMARY KEY,
  user_id BIGINT NOT NULL,
  source_service TEXT NOT NULL,
  event_id TEXT NOT NULL,
  type TEXT NOT NULL,
  actor_id BIGINT,
  entity_type TEXT,
  entity_id TEXT,
  body TEXT NOT NULL,
  read_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (source_service, event_id, user_id, type)
);
```

Questions:

1. Is `read_at` okay on this table?
2. What does the unique constraint protect against?
3. Should `body` be stored directly, or rendered from template data?
4. What index supports latest unread notifications?
5. What index supports retention cleanup?

## Part 4: Main query and index design

Main query:

```sql
SELECT id, actor_id, type, body, created_at
FROM notifications
WHERE user_id = $1
  AND read_at IS NULL
ORDER BY created_at DESC
LIMIT 50;
```

Candidate index:

```sql
CREATE INDEX idx_notifications_unread_user_created
ON notifications (user_id, created_at DESC)
WHERE read_at IS NULL;
```

Why this is probably the first index to try:

- `user_id` is the equality filter.
- `created_at DESC` supports the ordering.
- `WHERE read_at IS NULL` makes this a partial index containing only unread rows.
- `LIMIT 50` means the database can stop early after finding the newest unread notifications for that user.

But do not blindly trust the index. In the lab, we will test it with `EXPLAIN ANALYZE`.

## Part 5: Scaling pressure

Now change the assumptions:

- one user has 10 million notifications
- one celebrity event generates 20 million notifications
- badge count must be below 100 ms
- notification writes come from 20 services
- duplicate messages happen during retries

Revise the design by answering:

1. Does a single `notifications` table still work?
2. Do we need partitioning by `user_id`, time, or both?
3. Is badge count computed from the table or stored as derived data?
4. What happens when an event fans out to millions of users?
5. What is the source of truth?
6. What data can be eventually consistent?

## Part 6: Stream design

Design this flow:

```text
producer service -> event bus -> notification service -> database -> derived read models
```

Include:

- topic names
- partition key
- consumer group
- idempotency key
- retry behavior
- dead-letter behavior
- how to rebuild derived data

Example starting point:

```text
Topic: notification-events
Partition key: recipient_user_id or source aggregate id, depending on ordering needs
Consumer group: notification-writer
Database: Postgres as source of truth
Cache/read model: Redis badge count, eventually consistent
Deduplication: unique constraint on source_service + event_id + user_id + type
```

| Process incoming event |  |  |  |  |

Hints:

- Creating notifications usually benefits from idempotency and unique constraints.
- Marking one notification as read can usually be an idempotent update.
- Mark-all-read can be tricky for users with huge histories.
- Badge count can usually be eventually consistent.
- Pessimistic locking should be used carefully because it reduces concurrency.

## Part 8: Cloud sketch

Add a cloud deployment sketch.

Use generic names first; later we can map them to AWS/GCP/Azure.

Example:

```text
API service:
  handles notification reads and read-state writes

Managed Postgres:
  source of truth for notifications and read state

Managed stream:
  receives notification-worthy events from product services

Worker service:
  consumes stream events and writes notifications

Redis/cache:
  stores fast badge counts and maybe recent unread IDs

Object storage:
  optional, for large payloads or archived notification history

Observability:
  DB latency, stream lag, duplicate rate, cache hit rate, badge staleness, hot users
```

Cloud questions:

1. What runs in multiple availability zones?
2. Which reads can go to a read replica?
3. What breaks if the read replica lags?
4. What data is backed up?
5. What is the disaster recovery plan?
6. What is the biggest cost risk?
