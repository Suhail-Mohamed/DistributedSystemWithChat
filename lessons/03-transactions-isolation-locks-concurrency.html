How a database handles concurrent access

Transaction request → transaction manager assigns transaction/snapshot state → executor reads visible row versions → lock manager coordinates conflicting operations → writer creates or replaces a row version → WAL / durability machinery records the change → COMMIT makes the result durable and visible

MVCC / versions

Readers often inspect a snapshot and choose the row version visible to that transaction instead of blocking every writer.

Lock manager

Tracks incompatible operations, makes transactions wait, and helps detect deadlocks.

Validation

Optimistic and serializable schemes may allow work to proceed, then reject one transaction when a conflict is detected.

Retry loop

Serialization failures, deadlocks, failovers, and optimistic conflicts are normal outcomes that applications must handle safely.

Transaction locks vs internal latches

Transaction locks protect logical database objects and can be held until commit or rollback.

Latches/mutexes protect in-memory database structures for very short periods. They are an engine-internals mechanism, not an application transaction feature.

Isolation vs durability

Isolation controls what concurrent transactions may observe and which combinations may commit.

Durability controls whether committed work survives crashes. WAL and fsync are durability topics, not substitutes for isolation.

Read locks and write locks

ConceptGeneric meaningPostgreSQL anchorEffect
Shared/read lockMultiple readers may coexist, but incompatible writers wait.SELECT ... FOR SHARE is an explicit locking read.Useful when a later decision depends on the row remaining unchanged.
Exclusive/write lockA writer prevents incompatible changes to the same object.UPDATE, DELETE, and SELECT ... FOR UPDATE coordinate through row locks.Conflicting transactions wait, fail fast, or time out.
Predicate/range protectionProtects the result of a condition, including rows that do not exist yet.PostgreSQL Serializable tracks predicate-style read/write dependencies.Helps prevent phantoms and serialization anomalies.
Table lockCoordinates operations affecting an entire table or its definition.Ordinary commands automatically take table-level modes; strong DDL locks may block readers/writers.Essential for schema safety, but coarse locks reduce concurrency.
Important PostgreSQL detail: an ordinary SELECT normally does not take a row-level read lock. MVCC lets it read a snapshot while writers create newer row versions. Use FOR SHARE or FOR UPDATE only when the application needs a locking read.

Common PostgreSQL row-locking clauses

ClauseTypical intentStrength
FOR KEY SHAREProtect a referenced key from deletion or key-changing updates.Weakest row-locking clause.
FOR SHARETake a shared lock because a decision depends on the selected row.Allows compatible shared lockers; blocks conflicting updates.
FOR NO KEY UPDATEReserve a row for an update that does not change key identity.Stronger than share, weaker than full update lock.
FOR UPDATEReserve rows for a write-sensitive workflow.Strongest row lock among these clauses.

NOWAIT fails immediately rather than waiting. SKIP LOCKED skips locked rows and is useful for queue workers, but it deliberately returns an inconsistent view of the full table.

Isolation levels: standard idea vs PostgreSQL behavior

LevelSnapshot ideaPossible issuesPostgreSQL note
Read UncommittedWeakest standard level.Standard permits dirty reads.Behaves like Read Committed; PostgreSQL does not expose dirty reads.
Read CommittedFresh committed snapshot for each statement.Nonrepeatable reads, phantoms, application-level lost updates, serialization anomalies.Default PostgreSQL level.
Repeatable ReadStable transaction snapshot.Snapshot-isolation anomalies such as write skew can remain.PostgreSQL also prevents phantoms at this level, but conflicting writes may abort.
SerializableCommitted results must match some serial order.Transactions may be aborted and must be retried.PostgreSQL uses Serializable Snapshot Isolation and dependency detection.
Implementations vary. The SQL level name is not enough to predict every behavior across PostgreSQL, MySQL, SQL Server, Oracle, or distributed databases. Verify the product’s documented guarantees.

Concurrency anomalies to recognize

Dirty read

Read another transaction’s uncommitted data, which might later roll back.

Nonrepeatable read

Read one row twice and see different committed values during the same transaction.

Phantom

Repeat a predicate query and see a changed set of matching rows.

Lost update

Two clients derive writes from the same stale value, and one logical change is overwritten.

Write skew

Transactions read the same invariant, update different rows, and jointly violate the invariant.

Deadlock

A waits for B while B waits for A. The database aborts at least one transaction to break the cycle.

Pessimistic vs optimistic concurrency

ApproachHow it worksBest fitMain cost
PessimisticLock the row/resource before making a conflict-sensitive decision.High contention, short transactions, scarce resources such as the last ticket.Waiting, deadlocks, reduced concurrency, long lock hold times.
OptimisticRead a version, attempt a conditional write, and detect whether another writer won.Low-to-moderate contention and APIs where retry is straightforward.Wasted work and retries when conflicts are frequent.

Experiment 1 — Statement snapshot vs transaction snapshot

Read Committed

Session A
BEGIN ISOLATION LEVEL READ COMMITTED;
SELECT balance FROM accounts WHERE account_id = 1;
-- pause here
Session B
BEGIN;
UPDATE accounts
SET balance = 1200
WHERE account_id = 1;
COMMIT;

Back in Session A:

SELECT balance FROM accounts WHERE account_id = 1;
COMMIT;

Repeatable Read

Reset the lab, then repeat with Session A starting:

BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT balance FROM accounts WHERE account_id = 1;

Session B updates and commits as before. Session A re-reads the row from its stable snapshot, then commits.

Experiment 2 — Plain read vs locking read

Ordinary SELECT

Session A
BEGIN;
SELECT * FROM accounts WHERE account_id = 1;
-- keep transaction open
Session B
UPDATE accounts
SET balance = balance + 10
WHERE account_id = 1;

Finish Session A with COMMIT;.

SELECT FOR UPDATE

Reset the lab. Session A reserves the row:

BEGIN;
SELECT *
FROM accounts
WHERE account_id = 1
FOR UPDATE;

Session B sets a short timeout and attempts the conflicting write:

SET lock_timeout = '3s';
UPDATE accounts
SET balance = balance + 10
WHERE account_id = 1;
RESET lock_timeout;

Commit Session A, then retry the Session B update.

SELECT FOR SHARE

Reset the lab. Session A uses FOR SHARE. Session B can acquire another compatible share lock, but its update waits or times out until Session A ends.

Inspect locks while a session is blocked

While one session is waiting, run from a third terminal:

Experiment 3 — Lost update and optimistic version checks

Stale read-modify-write

Reset the lab. Both sessions read the same inventory row:

SELECT stock, version
FROM inventory
WHERE product_id = 1;

Session A writes the computed value and commits:

UPDATE inventory SET stock = 9 WHERE product_id = 1;

Session B later writes the same stale computed value:

UPDATE inventory SET stock = 9 WHERE product_id = 1;

Two logical decrements have produced only one stored decrement.

Optimistic version check

Reset. Both sessions read version 0. Each attempts:

UPDATE inventory
SET stock = stock - 1,
    version = version + 1
WHERE product_id = 1
  AND version = 0
  AND stock > 0
RETURNING product_id, stock, version;

Only one update should match version 0. The other client detects zero affected rows and must reread or retry.

Atomic update

UPDATE inventory
SET stock = stock - 1
WHERE product_id = 1
  AND stock > 0
RETURNING stock;

Encoding the check and modification in one SQL statement avoids an application-side read/write gap.

Experiment 4 — Phantom behavior

At Read Committed, Session A counts active bookings for a room and date range. Session B inserts a matching row and commits. Session A repeats the count inside its original transaction.

Reset and repeat with Session A using Repeatable Read. PostgreSQL’s stable snapshot keeps the repeated predicate result unchanged inside that transaction.

The exact statements are in labs/transactions-postgres/README.md.

Experiment 5 — Write skew under snapshot isolation

The doctors table starts with Alice and Bob both on call. The business rule says at least one doctor must remain on call.

Session A
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT count(*) FROM doctors WHERE on_call;
UPDATE doctors SET on_call = false
WHERE doctor_name = 'alice';
Session B
BEGIN ISOLATION LEVEL REPEATABLE READ;
SELECT count(*) FROM doctors WHERE on_call;
UPDATE doctors SET on_call = false
WHERE doctor_name = 'bob';

Commit both. They updated different rows, so both may commit even though the joint result violates the invariant.

Repeat at Serializable

Reset and use BEGIN ISOLATION LEVEL SERIALIZABLE; in both sessions. PostgreSQL should reject one transaction with a serialization failure. The application must retry the entire failed transaction.

Experiment 6 — Deadlock and lock ordering

Session A
BEGIN;
UPDATE accounts SET balance = balance - 10
WHERE account_id = 1;

-- after Session B locks account 2:
UPDATE accounts SET balance = balance + 10
WHERE account_id = 2;
Session B
BEGIN;
UPDATE accounts SET balance = balance - 20
WHERE account_id = 2;

-- after Session A waits for account 2:
UPDATE accounts SET balance = balance + 20
WHERE account_id = 1;

PostgreSQL detects the wait cycle and aborts one transaction. Roll back any failed transaction before continuing.

Prevention pattern: acquire locks in a consistent global order, such as lower account ID first. Still keep retry logic because deadlocks can arise through less obvious paths.

Optional experiment — Queue workers with SKIP LOCKED

Both sessions run:

BEGIN;
SELECT job_id, payload
FROM queue_jobs
WHERE status = 'ready'
ORDER BY job_id
FOR UPDATE SKIP LOCKED
LIMIT 1;

Each worker should reserve a different ready job without waiting. The worker updates its selected job to running and commits.

SKIP LOCKED is useful for work queues, not for general-purpose consistent reads. It intentionally omits rows locked elsewhere.

Application patterns

Last ticket

Use an atomic conditional update or a short FOR UPDATE transaction. Add idempotency around the purchase request.

Profile edit

A version column or ETag-style compare-and-swap often works well because conflicts are uncommon and retries are understandable.

Account transfer

Update both accounts in one transaction, lock them in a consistent order, preserve ledger/audit records, and retry deadlocks safely.

Business invariant

For rules spanning multiple rows, consider Serializable, explicit invariant locks, or redesigning the invariant around a single constrained row.

Cloud and distributed-system angle

  • Managed PostgreSQL keeps PostgreSQL semantics: moving to RDS, Cloud SQL, or managed Postgres does not remove lock waits, deadlocks, or isolation choices.
  • Network round trips lengthen transactions: extra application calls hold snapshots and locks longer.
  • Read replicas are a different topic: replica lag can violate read-your-writes even if the primary transaction used strong isolation.
  • Failover aborts in-flight work: clients need bounded retries and idempotency rather than assuming every timeout means failure.
  • Serverless bursts can overwhelm connections: use pooling and keep transactions short.
  • Cross-service workflows are not one local transaction: outbox, sagas, idempotency, and reconciliation handle wider workflows.

Common mistakes

Holding a lock during user input

Never keep a transaction open while waiting for a human or remote service when a reservation/state-machine design will work.

Retrying only the failed statement

A serialization failure invalidates the transaction’s assumptions. Retry the complete transaction.

Treating timeout as rollback proof

A client timeout may occur after the server committed. Use idempotency keys and verify outcomes.

Using stronger isolation blindly

Serializable can simplify correctness, but workloads need retry handling, short transactions, and monitoring.

Assuming ordinary SELECT locks rows

In PostgreSQL, normal reads use snapshots; explicit locking reads must be requested.

Ignoring constraints

Unique constraints, checks, and foreign keys are concurrency tools. Application checks alone are race-prone.

What mastery looks like

  • You can trace what each session sees under Read Committed, Repeatable Read, and Serializable.
  • You know when a normal read uses MVCC versus when a locking read is required.
  • You can choose between atomic SQL, optimistic version checks, pessimistic locks, and Serializable.
  • You expect deadlocks and serialization failures and design safe retries.
  • You separate transaction isolation from replication lag and distributed workflows.
