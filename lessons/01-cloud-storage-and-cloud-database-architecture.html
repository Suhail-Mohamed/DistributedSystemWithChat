Things you should know first

1. Object storage is not a database

Object storage stores bytes behind object keys. It is great for logs, media, backups, and data-lake files. By itself, it does not give you SQL planning, indexes, transactions, constraints, or joins.

2. OLTP and OLAP optimize for different work

OLTP systems care about low-latency writes, point reads, constraints, and transactions. OLAP systems care about scans, compression, aggregation, joins, column pruning, and large analytical queries.

3. Cloud warehouses often separate compute and storage

Storage can live in durable cloud storage while compute clusters scale independently to run queries.

4. Metadata is a real system component

Catalogs, table schemas, statistics, permissions, file lists, micro-partition metadata, and query coordination often live in a control-plane layer.

Mental model

Use this whenever looking at a cloud data system:

Application or analyst
→ SQL/API endpoint
→ cloud services / metadata / optimizer
→ compute workers / virtual warehouse
→ files, objects, pages, partitions, or shards

The most important habit: ask whether the system is optimized for serving application requests, processing events, or analyzing large datasets.

Snowflake-style anchor

For this course, treat Snowflake as a useful architecture example:

  • Database storage: compressed columnar table data in cloud storage.
  • Compute: virtual warehouses that process SQL.
  • Cloud services: authentication, metadata, query parsing, optimization, and coordination.
  • Micro-partitions: automatic storage units with metadata that can be pruned during queries.

Verification checklist

A strong answer should include these claims in your own words:

  • Object storage is not a replacement for a transactional database.
  • Cloud analytic databases often separate compute, storage, and metadata/control-plane work.
  • Micro-partitioning/pruning is not the same thing as a regular B-tree index.
  • Storage layout affects query cost.
  • Analytics systems optimize scans, compression, column pruning, and partition pruning.
  • OLTP systems optimize point lookups, constraints, indexes, and transactions.
  • Cloud design should include cost, reliability, observability, and failure modes.
