Things you should know first

Partitioning

Split one logical dataset into smaller pieces according to a rule. The pieces may remain on one database server or be distributed.

Partition key

The value used to decide where a record belongs. Its quality depends on both data distribution and access patterns.

Partition pruning

Exclude partitions that provably cannot match a query. Pruning avoids work; it does not magically make every query fast.

Sharding

Partition data across multiple machines or database instances. Sharding adds routing, failures, rebalancing, and cross-shard operations.

Replication

Keep copies of the same data for availability or read scale. Partitioning divides data; replication copies it.

Hot partition

One partition receives much more traffic or storage than others. A mathematically even hash does not fix a single extremely hot key.

Partitioning is not automatically distributed

One PostgreSQL instance logical table: events ├── events_2026_01 ├── events_2026_02 └── events_2026_03 All pieces can still live on one server.

Sharding adds network boundaries

Application / router ├── shard A → users 0–9999 ├── shard B → users 10000–19999 └── shard C → users 20000–29999 Now routing and cross-shard work matter.

The five-part DDIA-style checklist

1. PlacementWhich key/rule places a record?
2. RoutingHow does a request find the right piece?
3. SkewCan one key or range dominate?
4. RebalancingWhat moves when capacity changes?
5. IndexesAre secondary lookups local or scattered?

This course uses DDIA for its comparative, tradeoff-first style—not as a chapter reproduction.

Three common partitioning methods

MethodPlacement ruleStrengthMain riskTypical use
RangeKey falls between lower and upper bounds.Excellent for range queries and lifecycle operations.Sequential keys can concentrate new writes in one range.Events by month, orders by date, numeric ID ranges.
Hashhash(key) mod N or a token/slot scheme.Usually spreads distinct keys more evenly.Destroys useful ordering and cannot split one hot key by itself.User/session placement, distributed KV records.
ListExplicit values map to named partitions.Easy operational control by region, tier, or category.Uneven categories and ongoing manual maintenance.Region, regulatory zone, tenant class.
Partition key choice is workload design. A key can be excellent for one query and poor for another. A date key helps time-range scans but not user lookups. A user hash helps equality routing but not date-range pruning.

Parquet data lakeFolders/files/row groups.You or a table format/ingestion job controls layout.Path, file, and row-group metadata enable skipping.Too many tiny files can become an operational problem.

Practical product examples

Event retention

Range partitions by month make recent-window scans and old-data removal natural. The newest partition may receive nearly all writes.

User sessions

Hashing by user ID spreads many users well and supports direct equality routing. A single celebrity or bot key can still become hot.

Chat messages

Partitioning by conversation keeps message history together. A huge group chat can overwhelm one partition, so the design may need time buckets.

Multi-tenant SaaS

Tenant ID provides isolation and tenant-local queries. One enterprise tenant can dwarf all others, creating storage and throughput skew.

Orders

Customer-based placement helps “my orders”; order-ID placement helps direct order lookup. Reporting by time may require another index or derived system.

Metrics

Time partitions support retention, while hashing within a time window can distribute writes. This is a common reason for two-level partitioning.

Lab C — Hot partition simulator

The simulator compares a uniform workload, a workload dominated by one hot key, and a write-sharded version of that hot key.

python labs/partitioning-postgres/hot_partition_sim.py

The output prints request counts per partition and the busiest partition’s share of traffic.

Key detail: adding more partitions spreads many independent keys, but one unchanged hot key still hashes to one destination. Write sharding can spread it, but reads and aggregation become more complicated.

Common partitioning mistakes

Using a low-cardinality key

Values such as status = pending/completed often create only a few uneven pieces.

Assuming partitioning replaces indexes

Pruning chooses relevant partitions; indexes can still locate rows inside those partitions.

Creating too many pieces

Planner time, metadata, file counts, connection pools, and operational complexity can rise.

Ignoring the hottest key

Uniform overall distribution can hide one customer, tenant, or item consuming most capacity.

Forgetting cross-partition constraints

Uniqueness and foreign-key rules may be harder to enforce when data is split across nodes.

No rebalance plan

A design is incomplete if adding nodes requires a full outage or moving nearly all data at once.

What mastery looks like

  • You distinguish partitioning, sharding, replication, clustering, pruning, and indexing.
  • You choose a key from access patterns and expected skew—not from column names alone.
  • You can explain why range and hash strategies help different queries.
  • You inspect query plans to verify pruning instead of assuming it occurred.
  • You include routing, rebalancing, secondary indexes, and hot keys in a distributed design.
