Course Format and Roadmap

One topic per lesson. Each lesson teaches the background, then asks you to experiment, answer questions in one place, and submit a Markdown artifact for review.

Lesson shape

1. What you should know

Definitions, mental models, product examples, and details that commonly cause confusion.

2. Try the technology

A focused experiment using DuckDB, PostgreSQL, Docker, Redis, or a stream system.

3. Use one question file

All predictions, observations, and design prompts live under questions/.

4. Commit answers

Write a separate Markdown answer file in the lab folder, commit it, and ask for review in chat.

Design influence

This course uses the comparative, tradeoff-first style of Designing Data-Intensive Applications: understand the fundamental mechanism, then compare how real systems make different choices.

For concurrency topics, separate snapshots, locks, validation, invariants, and retry behavior. A transaction isolation label alone does not explain every product’s implementation.

Review rubric

AreaWhat I will check
Access patternsDid the design start with reads/writes instead of technology names?
Data modelDoes the schema/key model support the product behavior?
Indexes and plansDo indexes and physical plans match the query shapes?
Partitioning/shardingDoes the placement strategy address routing, skew, and rebalancing?
CorrectnessAre snapshots, locks, retries, idempotency, and consistency handled?
Cloud realismDoes the design mention managed services, regions/AZs, cost, and observability?
Failure modesDoes the design admit what breaks and how recovery works?
